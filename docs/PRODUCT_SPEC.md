## Product Summary
Школьный ассистент помогает родителям и ученикам 1-7 классов разбирать школьные задания простым, спокойным и наглядным языком. MVP строится вокруг быстрой пользы без регистрации: пользователь может сразу задать вопрос текстом, отправить фото задания или голосовое сообщение, получить 1-2 качественных ответа и только после этого увидеть мягкий сценарий capture.

Главная бизнес-цель MVP: собрать родительскую аудиторию через реальную бесплатную ценность, затем перевести ее в сохраненные аккаунты и удерживать через историю, фильтры и повторные сценарии использования.

## Business Goals
1. Сформировать базу родителей как primary audience.
2. Собирать качественные контакты после первого полезного опыта, а не до него.
3. Повышать повторное использование через сохраненную историю и понятную структуру чатов.
4. Запустить полностью бесплатную фазу на 1-3 месяца.
5. После бесплатной фазы ввести оплату по модели, близкой к себестоимости, без агрессивной монетизации.

## Target Users and Roles
### Guest
Гость входит без регистрации и может:
- задать текстовый вопрос;
- загрузить фото задания;
- отправить голосовое сообщение;
- получить ограниченное число ответов;
- увидеть предложение сохранить прогресс и открыть историю.

Гость не может:
- пользоваться полной историей;
- иметь долгосрочное хранение прогресса;
- получать расширенные лимиты.

### Parent
Родитель является primary user MVP и может:
- вести одного ребенка;
- указать класс ребенка;
- получать объяснения, понятные ребенку;
- получать отдельные подсказки для родителя;
- сохранять и фильтровать историю;
- в будущем покупать пакеты запросов.

### Student
Ученик является secondary user и может:
- пользоваться сервисом самостоятельно;
- зарегистрироваться через email;
- указать возраст;
- получать понятные объяснения по школьной программе.

### Admin
Администратор управляет:
- лимитами;
- prompt profiles по классам;
- feature flags;
- пользователями и сегментами;
- usage и cost-метриками;
- broadcast-кампаниями;
- ручным начислением запросов;
- тарифами и trial-периодом.

## MVP Scope
### In Scope
- guest-first вход без регистрации;
- один основной сценарий: задать школьный вопрос и получить полезный разбор;
- три входных модальности: text, image, voice;
- prompt-профили по классам;
- история чатов для зарегистрированных пользователей;
- группировка истории по классу, предмету, теме и чатам;
- фильтры по дате, месяцу, предмету, теме и роли;
- capture после value moment;
- admin-панель для лимитов, пользователей, промптов, cost/usage и рассылок;
- базовый платежный контур для будущей cost-based монетизации.

### Out of Scope
- загрузка учебников в базу;
- отдельный OCR-пайплайн собственного уровня;
- multi-child аккаунты;
- кабинеты школ;
- Obsidian export;
- учебный календарь и полноценные study plans;
- сложная аналитика успеваемости;
- marketplace репетиторов;
- Telegram как основной способ входа.

## Academic Scope
- Основной фокус: 1-7 классы.
- Предметный охват: все основные школьные предметы в рамках школьной программы.
- Ограничение AI-ответов: не уходить в университетский уровень без запроса, не превращать продукт в универсальный чат "обо всем".

## Core Value Proposition
Каждый ответ должен:
- объяснять материал простыми словами;
- показывать логику, а не только финальный ответ;
- по необходимости давать пошаговый разбор;
- по возможности предлагать 2-3 пути решения;
- в parent-help mode подсказывать родителю, как помочь ребенку самому.

Для математических выражений требуется точный рендер через математический движок. Генеративные изображения для точной математики в MVP не используются.

## Funnel
### Entry
- Вход в режиме guest.
- Минимальное трение.
- Базовые экраны входа: landing и chat.
- Примеры CTA: "Объяснить тему простыми словами", "Разобрать фото из учебника", "Помочь с домашним заданием".

### Value Moment
- До capture пользователь должен получить 1-2 полезных ответа.
- Value moment считается достигнутым, если родитель чувствует экономию времени, снижение стресса, а ребенок понимает материал лучше, чем после чтения учебника.

### Capture
Capture запускается только после пользы и при одном из триггеров:
- сохранение чата;
- открытие истории;
- попытка продолжить после лимита;
- попытка разблокировать расширенное фото или voice использование;
- запрос родительской памятки.

Данные capture:
- для родителя: имя, роль, класс ребенка, email, опционально телефон;
- для ученика: роль, возраст, email;
- обязательное согласие на обработку персональных данных;
- отдельное согласие на маркетинговые коммуникации.

### Retention
Retention строится на:
- сохраненной истории;
- повторном входе по предмету;
- фильтрах по темам;
- продолжении прошлых диалогов;
- сообщениях и рассылках для родителей.

### Monetization
- Фаза запуска: полностью бесплатный доступ на 1-3 месяца.
- Следующая фаза: cost-based access с пакетами `10_days`, `50_requests`, `100_requests`.
- Будущие Pro-надстройки есть в roadmap, но не входят в MVP.

## Core User Flows
### Guest Learning Flow
1. Пользователь открывает landing или сразу чат.
2. Выбирает текст, фото или голос.
3. Получает полезный ответ в одном из response modes.
4. После 1-2 ответов или при попытке продолжить сверх лимита видит capture CTA.

### Parent Saved Flow
1. Родитель завершает capture.
2. Сервис создает сохраненный аккаунт и связывает с ним историю.
3. Родитель возвращается в историю, фильтрует чаты и продолжает диалоги.

### Student Self-Serve Flow
1. Ученик проходит упрощенный capture с ролью student.
2. Указывает возраст.
3. Получает объяснения, адаптированные под возрастной уровень.

### Admin Operations Flow
1. Админ отслеживает usage, конверсии и стоимость.
2. Меняет квоты, prompt profiles и feature flags.
3. Управляет аудиториями и рассылками.

## Response Modes
MVP должен поддерживать следующие режимы:
- simple_explanation;
- step_by_step_solution;
- answer_with_reasoning;
- multiple_solution_paths;
- parent_help_mode.

Пользовательские названия режимов могут быть локализованы, но смысл должен оставаться тем же.

## Limits and Access Rules
- Все лимиты управляются из админки.
- Лимиты считаются отдельно по text/image/voice usage.
- Превышение лимита ведет в мягкий CTA, а не в жесткую блокировку без объяснения.
- У guest лимиты ниже, чем у зарегистрированного free пользователя.
- История и долгосрочное хранение прогресса являются retention-ценностью для зарегистрированных пользователей.

## Admin Requirements
Админка MVP должна поддерживать:
- обзор ключевых продуктовых и cost-метрик;
- управление квотами;
- редактирование prompt profiles по классам;
- просмотр пользователей и сегментов;
- отправку контентных сообщений и broadcast-кампаний;
- ручное начисление запросов;
- управление abuse cases;
- просмотр платежей и тарифов;
- экспорт контактов.

## Open Questions
1. В JSON-спеке диапазон классов указан как `1_to_6_or_1_to_7`, а в MVP ТЗ как `1-7`. В документе принят диапазон `1-7`, так как он явно зафиксирован в ТЗ, но это нужно окончательно подтвердить.
2. В JSON-спеке для родителя в capture перечислен `email` и `phone_optional`, а в ТЗ фигурирует `email или телефон`. Нужна финальная бизнес-правка: обязателен ли email для parent capture или допустим любой один контакт.
3. `get_parent_cheatsheet` указан как trigger capture в JSON-спеке, но сама сущность или экран parent cheatsheet отдельно не описаны. Сейчас это трактуется как CTA/offer, а не самостоятельная MVP-функция.
4. История в JSON-спеке фильтруется также по `role`, а в ТЗ в экранной части перечислены только дата, предмет и тема. В продуктовой спецификации роль оставлена как допустимый фильтр, но стоит подтвердить, нужен ли он в первом UI-релизе.
